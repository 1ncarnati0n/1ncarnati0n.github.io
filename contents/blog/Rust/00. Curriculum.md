## Phase 1: Foundation (Day 1-13)

> Rust의 핵심 철학과 메모리 안전성의 기반을 세우는 단계

### Day 1-2: Setup & Basic Syntax

| 주제    | 핵심 개념                                              |
| ----- | -------------------------------------------------- |
| Cargo | `new`, `build`, `run`, `check` 워크플로우               |
| 변수    | 기본 immutable, `mut` 명시, shadowing (타입 변경 가능)       |
| 타입    | 정수(`i32`), 실수(`f64`), `bool`, `char`, 튜플, 배열       |
| 제어 흐름 | `if`/`else`, `loop`, `while`, `for`, 모두 expression |

**Python과의 핵심 차이:** Rust는 기본 immutable (Python은 기본 mutable)

### Day 3-5: Ownership

| 주제      | 핵심 개념                                      |
| ------- | ------------------------------------------ |
| 소유권 3원칙 | 값마다 소유자 하나, 스코프 끝나면 drop, 한 번에 하나          |
| Move    | Heap 타입(`String`, `Vec`) 대입 시 소유권 이동       |
| Copy    | Stack 타입(`i32`, `bool`, `char`) 대입 시 비트 복사 |
| Clone   | Heap 타입의 명시적 깊은 복사 (`.clone()`)            |
| 함수 전달   | 함수에 값을 넘기면 소유권도 이동 (Move semantics)        |

**핵심 통찰:** double-free 방지를 위한 컴파일 타임 보장

### Day 6-8: References & Borrowing

| 주제 | 핵심 개념 |
|------|----------|
| 핵심 규칙 | `&T` 여러 개 OR `&mut T` 하나 (동시 공존 불가) |
| 목적 | data race를 컴파일 타임에 방지 |
| NLL | Non-Lexical Lifetimes — 참조의 수명은 마지막 사용 시점까지 |
| 최소 권한 | 읽기만 하면 `&`, 수정 필요하면 `&mut` |

**Python과의 핵심 차이:** Python `참조`는 항상 수정 가능, Rust `&`는 기본 읽기 전용

### Day 9-10: Structs & Methods

| 주제 | 핵심 개념 |
|------|----------|
| struct | 데이터 정의 (Python `class`의 필드 부분) |
| impl | 동작 정의 (데이터와 분리) |
| self 3종 | `&self`(읽기), `&mut self`(수정), `self`(소비) |
| Associated fn | `Self::new()` — `self` 없이 호출 (`::` 사용) |
| Struct update | `..other`로 나머지 필드 복사 (Move 주의) |

**핵심 통찰:** `&self` = 소유권 보존, `self` = 소유권 소비

### Day 11-13: Enums & Pattern Matching

| 주제 | 핵심 개념 |
|------|----------|
| enum | variant마다 다른 데이터를 담을 수 있음 |
| match | exhaustiveness — 모든 variant 처리 필수 |
| Option\<T\> | `Some(T)` / `None` — Rust의 null 대체 |
| Result\<T, E\> | `Ok(T)` / `Err(E)` — 에러 처리의 기본 |
| if let | 하나의 패턴만 처리할 때 match의 간결한 대안 |
| Combinators | `map`, `and_then`, `unwrap_or`, `ok_or` 등 |

**핵심 통찰:** `Option`/`Result` = 안전한 값 처리의 표준 패턴

## Phase 2: Data & Functional (Day 14-20)

> 컬렉션, 문자열 내부 구조, 함수형 프로그래밍 패턴을 익히는 단계

### Day 14-15: Collections

| 주제 | 핵심 개념 |
|------|----------|
| Vec\<T\> | 동적 배열, `push`/`pop`/`get`/인덱싱 |
| HashMap\<K, V\> | 키-값 저장소, `insert`/`get`/`entry` API |
| entry API | `or_insert`, `and_modify` — 조건부 삽입/수정 |
| Ownership | 컬렉션에 값을 넣으면 소유권 이동 |
| Borrowing 충돌 | `&v[0]` 살아있을 때 `v.push()` 불가 (재할당 위험) |

**Python 대응:** `list` → `Vec`, `dict` → `HashMap`

### Day 16-17: Slices 심화 & String 내부

| 주제 | 핵심 개념 |
|------|----------|
| `&[T]` / `&str` | 슬라이스 = fat pointer (포인터 + 길이) |
| String vs &str | `String`(소유, Heap) vs `&str`(빌림, 어디서든) |
| UTF-8 | Rust 문자열은 UTF-8, 인덱싱 불가 (`s[0]` ❌) |
| 문자열 접근 | `.chars()`, `.bytes()`, `.char_indices()` |
| 변환 | `&String` → `&str` (자동 deref), `&str` → `String` (`.to_string()`) |

**핵심 통찰:** fat pointer = (데이터 포인터, 길이) — 안전한 메모리 접근의 핵심

### Day 18-20: Iterators & Closures

| 주제 | 핵심 개념 |
|------|----------|
| Iterator trait | `.next()` → `Option<Item>`, lazy evaluation |
| 어댑터 | `map`, `filter`, `enumerate`, `zip`, `take`, `skip`, `chain` |
| 소비자 | `collect`, `sum`, `count`, `any`, `all`, `find`, `fold` |
| Closures | `\|args\| body`, 환경 캡처 (`&`, `&mut`, move) |
| Fn traits | `Fn`(불변 빌림), `FnMut`(가변 빌림), `FnOnce`(소비) |
| 소유권 연결 | `.iter()`(`&T`), `.iter_mut()`(`&mut T`), `.into_iter()`(`T`) |

**Python 대응:** list comprehension → `.iter().filter().map().collect()`

## Phase 3: Type System & Error Handling (Day 21-28)

> Rust 타입 시스템의 핵심 — Traits, Generics, Lifetimes로 추상화 능력을 확장

### Day 21-22: Error Handling 심화

| 주제 | 학습 내용 |
|------|----------|
| `?` 연산자 | 에러 자동 전파 — `match`/`unwrap` 대신 간결한 에러 처리 |
| 커스텀 에러 | `enum`으로 도메인 에러 타입 정의 |
| From trait | `?`가 에러 타입을 자동 변환하는 메커니즘 |
| 에러 전략 | `panic!` vs `Result` — 언제 어떤 것을 쓸지 |
| 실전 패턴 | 파일 I/O, 파싱 등에서의 에러 처리 체인 |

**선수 지식:** Phase 1의 `Result<T, E>`, `Option<T>`, enum, pattern matching

### Day 23-26: Traits & Generics

| 주제 | 학습 내용 |
|------|----------|
| Trait 정의 | `trait` = 인터페이스 — 공유 행동 정의 |
| Trait 구현 | `impl Trait for Type` |
| 표준 Traits | `Display`, `Debug`, `Clone`, `PartialEq`, `Iterator` 등 |
| derive 매크로 | `#[derive(Debug, Clone, PartialEq)]` 자동 구현 |
| Generics | `fn process<T>(item: T)` — 타입 파라미터 |
| Trait Bounds | `<T: Display + Clone>` — 제네릭에 제약 추가 |
| where 절 | 복잡한 bounds를 깔끔하게 표현 |
| impl Trait | 함수 인자/반환에서의 trait 활용 |

**Python 대응:** ABC(Abstract Base Class) → `trait`, duck typing → trait bounds

### Day 27-28: Lifetimes

| 주제 | 학습 내용 |
|------|----------|
| Lifetime이란 | 참조가 유효한 범위를 명시하는 어노테이션 |
| `'a` 문법 | `fn longest<'a>(x: &'a str, y: &'a str) -> &'a str` |
| Lifetime elision | 컴파일러가 자동 추론하는 3가지 규칙 |
| Struct의 lifetime | 참조를 담는 struct에 `'a` 필요 |
| `'static` | 프로그램 전체 수명 동안 유효한 참조 |

**핵심 목표:** "dangling reference 방지"를 컴파일러가 어떻게 보장하는지 이해

## Phase 4: Advanced & Ecosystem (Day 29-34)

> 힙 메모리 관리와 동시성의 기초 — Rust의 고급 메모리 패턴

### Day 29-30: Smart Pointers

| 주제 | 학습 내용 |
|------|----------|
| Box\<T\> | 힙 할당, 재귀 타입, trait object |
| Rc\<T\> | 참조 카운팅 — 여러 소유자 (싱글 스레드) |
| Arc\<T\> | 원자적 참조 카운팅 (멀티 스레드) |
| RefCell\<T\> | 런타임 borrow checking — interior mutability |
| Deref trait | 스마트 포인터의 자동 역참조 |

**Python 대응:** Python의 모든 객체가 사실상 `Rc<RefCell<T>>`와 비슷한 구조

### Day 31-32: Concurrency 기초

| 주제 | 학습 내용 |
|------|----------|
| std::thread | 스레드 생성과 `join` |
| move closure | 클로저가 환경을 소유권으로 캡처 |
| Send / Sync | 스레드 간 데이터 전송 안전성 trait |
| Mutex\<T\> | 상호 배제 — `lock()` → `MutexGuard` |
| Arc + Mutex | 멀티 스레드 공유 데이터의 표준 패턴 |
| 채널 | `mpsc::channel` — 메시지 패싱 |

**핵심 목표:** "fearless concurrency" — 컴파일 타임에 data race 방지

### Day 33-34: Type Conversions & Common Patterns

| 주제 | 학습 내용 |
|------|----------|
| From / Into | 타입 변환의 표준 trait |
| TryFrom / TryInto | 실패 가능한 변환 (→ `Result`) |
| AsRef / AsMut | 저렴한 참조 변환 |
| Display / ToString | 문자열 표현 |
| Iterator 심화 | 커스텀 Iterator 구현 |
| Builder 패턴 | 복잡한 구조체 생성 패턴 |
| Newtype 패턴 | 타입 안전성을 위한 래퍼 |

## Phase 5: Project Infrastructure (Day 35-38)

> 실전 프로젝트를 위한 구조화, 테스팅, 생태계 활용

### Day 35-36: Modules, Testing & Project Structure

| 주제 | 학습 내용 |
|------|----------|
| mod 시스템 | `mod`, `use`, `pub` — 코드 구조화 |
| 파일 구조 | `lib.rs` vs `main.rs`, 모듈 트리 |
| 가시성 | `pub`, `pub(crate)`, `pub(super)` |
| #[test] | 단위 테스트, `assert!`, `assert_eq!` |
| #[cfg(test)] | 테스트 모듈 분리 |
| Integration test | `tests/` 디렉토리, 통합 테스트 |
| Documentation | `///` doc comment, `cargo doc` |

**Python 대응:** `import` → `use`, `__init__.py` → `mod.rs`

### Day 37-38: 실전 크레이트 생태계

| 크레이트 | 용도 | Python 대응 |
|----------|------|------------|
| serde + serde_json | 직렬화/역직렬화 | `json` 모듈 |
| clap | CLI 인자 파싱 | `argparse` |
| thiserror | 라이브러리용 에러 정의 | 커스텀 Exception |
| anyhow | 애플리케이션용 에러 처리 | 일반 Exception 처리 |
| tokio (소개) | 비동기 런타임 | `asyncio` |

**핵심 목표:** `Cargo.toml` 의존성 관리 + 실전에서 자주 쓰는 크레이트 체험

## Phase 6: Advanced Concepts (Day 39-40)

> Rust의 메타프로그래밍과 안전성 경계를 이해하는 마무리 단계

### Day 39-40: Macro 기초 & Unsafe 인식

| 주제         | 학습 내용                                     |
| ---------- | ----------------------------------------- |
| 선언적 매크로    | `macro_rules!` — `vec![]`, `println!`의 원리 |
| 매크로 패턴     | 반복(`$()*`), 매칭, 토큰 트리                     |
| 절차적 매크로    | derive macro 개념 이해 (깊이 파지는 않음)            |
| unsafe 키워드 | 안전한 Rust가 할 수 없는 5가지                      |
| unsafe 사용처 | raw pointer, FFI, `unsafe impl`           |
| 철학         | unsafe = "내가 안전성을 보장하겠다"는 선언              |


**핵심 목표:** 매크로의 동작 원리 이해 + unsafe의 존재 이유와 사용 기준 인식
